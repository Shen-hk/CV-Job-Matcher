# TieLink 业务逻辑整改任务清单

> 维护规则：每次调整业务链路后，先更新这里，再改代码。  
> 状态说明：`已完成` / `进行中` / `未开始`

## 1. 当前结论

当前项目已经从“多入口分散跳转”开始收拢到“聊天主入口 + 操作台式业务页”的方向，但整体仍处于半重构状态。

已明确的主链路是：
- 聊天页为主入口
- JD / 简历 / 跟进是能力模块，不再各自独立成孤岛
- 跟进页需要承接 JD 上下文，而不是空页跳转

## 4. 未开始项

| 状态 | 问题 | 哪里乱 / 缺了什么 | 带来的影响 | 建议做法 |
|---|---|---|---|---|
| 未开始 | 业务状态没有统一中心 | 登录、加载、编辑、提交、保存等状态分散在各页面 | 用户体验不一致，后续容易重复实现 | 建一个统一状态中心，沉淀公共状态字段和事件 |
| 未开始 | 异常链路兜底不完整 | 网络失败、权限拒绝、会话过期、空数据场景还不完整 | 用户会卡住，不知道下一步做什么 | 补齐兜底页、重试按钮、降级提示 |
| 未开始 | 业务规则缺少统一标准 | 不同页面对“是否可改、是否确认、何时保存”规则不一致 | 用户容易困惑，产品规则不稳定 | 输出统一业务规则表，所有页面按同一标准执行 |
| 未开始 | 流程评审机制不固定 | 新功能容易直接加入口，容易再次变乱 | 后续越做越散 | 新增业务必须先过流程评审和状态校验 |
| 未开始 | 合规/权限时机未统一 | 权限申请、数据记录、用户确认时机还没标准化 | 容易造成打扰或合规风险 | 统一权限弹窗时机、记录策略、隐私提示 |

## 5. 本轮整改重点

### 已先修的主问题

1. 聊天页到跟进页的链路断层
2. 跟进页新增表单无法承接 JD 上下文
3. 跟进状态切换逻辑过于粗糙

### 下一轮优先补的主问题

1. 卡片式确认
2. 自动保存提示
3. result 页中转标准化
4. 统一业务状态中心

---

## 6. 交付判定标准

当下面 4 条都满足时，才算“跟进链路”基本闭环：

- 从聊天页进入相关业务页，不会丢上下文
- 从 JD 列表进入跟进页，会自动预填可执行信息
- 关键动作有确认卡片，不是裸点即执行
- 保存结果有明确反馈，失败可重试，成功可回退
