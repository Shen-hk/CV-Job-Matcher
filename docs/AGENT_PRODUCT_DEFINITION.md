# TieLink Agent 产品定义 1.0

> 目标：把 TieLink 定义成一个“求职操作台型 Agent”，主入口是聊天界面，围绕求职闭环持续推进，而不是普通泛聊天。

---

## 1. 产品定位

- 主入口：聊天界面
- 产品形态：求职操作台型 Agent
- 第一责任：执行和推进求职动作
- 核心能力：识别意图、主动追问、调用工具、给出可确认结果、沉淀上下文

### 产品心智

- 用户用一句话发起需求
- Agent 负责判断、执行、提醒、确认、沉淀
- 聊天不是终点，而是业务推进的起点

---

## 2. 目标用户

- 主目标用户：应届生
- 次目标用户：所有求职人群
- 服务范围：简历、JD、面试、投递、跟进、打招呼等求职相关动作

### 用户范围原则

- 业务对象可以覆盖更广，但产品心智优先围绕应届生设计
- 所有能力必须回到“求职闭环”，不做泛化闲聊产品

---

## 3. 角色口吻

- Agent 口吻：顾问型
- 风格要求：专业、清晰、可信、少废话
- 交互方式：像懂求职流程的顾问，不像泛聊天机器人

### 推荐语气

- 先判断，再建议
- 先确认，再修改
- 先推进，再沉淀

---

## 4. 核心工作方式

1. 用户一句话发起需求
2. Agent 先判断意图
3. 信息不全时主动追问
4. 信息足够时自动执行
5. 需要修改时先确认
6. 结果以文本 + 卡片形式呈现
7. 用户确认后继续推进下一步

### 自动化原则

- 可以自动做：解析、推荐、提醒、生成初稿、归档上下文
- 必须确认后再做：修改简历内容、生成正式结果、覆盖已有内容、自动保存重要业务信息
- 原则：所有自动动作都要可见、可撤销、可追溯

---

## 5. 首发主场景

- JD 分析
- 简历优化
- 模拟面试
- 投递追踪

### 扩展原则

- 其他能力可以加，但不能抢主链路
- 新工具必须符合现有上下文规范
- 新页面必须能回流到聊天主链路

---

## 6. 卡片规范

- 需要用户操作、确认、上传、修改的内容统一用卡片展示
- 文本气泡负责对话
- 卡片负责操作
- 结果页可作为预览中转页

### 卡片职责

- 选择
- 确认
- 预览
- 对比
- 一键执行

### 展示原则

- 纯文本负责解释
- 卡片负责动作
- 能做成“气泡 + 卡片嵌套”的，优先这样做

---

## 7. 首页默认策略

- 不固定死入口按钮逻辑
- 根据上下文自动进入业务状态

### 典型规则

- 没有 JD 和简历时，先引导开始分析
- 有 JD 没简历时，优先补简历
- 有 JD 和简历时，优先给匹配和优化建议
- 有投递记录时，优先提示跟进

---

## 8. 页面闭环

- 起点：聊天页
- 中转：预览页 `result`
- 终点：回到聊天页继续推进

### 页面角色

- 聊天页：业务入口和推进中枢
- `result` 页：确认、预览、回看中转页
- 不把 `result` 定义为终点页

---

## 9. 上下文范围

- 当前 JD
- 当前简历版本
- 当前投递记录
- 当前面试会话
- 当前结果页状态
- 历史对话摘要

### 记忆原则

- 记忆只保留可复用信息
- 不做无边界堆积
- 按场景注入上下文，不把所有信息一次性塞进模型

---

## 10. 业务原则

- 主入口只有一个
- 所有能力都能被自然语言触发
- 所有关键动作都有确认和兜底
- 所有业务对象有唯一真相源
- 所有页面都服务于求职闭环

---

## 11. 推荐的产品边界

- 这不是一个泛聊天机器人
- 这不是一个纯工具集合
- 这是一个围绕求职任务推进的对话式操作台

### 最终形态

- 用户说需求
- Agent 追问补齐
- Agent 调用工具
- Agent 用卡片展示结果
- 用户确认后继续推进下一步
